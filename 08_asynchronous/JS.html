Control Flow

A program can behaves based on conditions instead of just executing the code line by line: this is control flow. The ability to control the code flow could be done in 3 ways but always evaluating a condition. A condition is made up of the following concept: <value> <comparison-operator> <value> The result of that comparison will condition the code.

Four ways to handle conditions

Condition kind and Syntax Note

if [ [else if] else ] condition

if( <condition> ){ /*...*/ }
if( <condition> ) { /*...*/ }
else { /* ... */ }
if( <condition> ){ /* ... */ }
else if (<condition>) { /* ... */ }
else { /* ... */ }
  • efficient and flexible and common control flow
  • can check by evaluating only one function with just a if condition statement
  • can check 2 to more conditions with if else if statements
  • can check 2 and one default conditions with if else if else ( else being considered as default) statements

switch condition

switch(<value>) {
  case <value-to-check>:
     /* ... */
  case <value-to-check>:
     /* ... */
  default:
     /* ... */
}
  • useful when one want to check the value equality and describe different code case scenario

ternary condition

<condition>
 ? <what-to-do-if-true>

  • handy for quick checks
  • handy to conditionally assign a value for instance

Logical Operators

Logical operators are great to combined with condition and control flow: supporting, but also for assigning a value to a variable either:

  • or one condition including multiple values to be checked ("and" operator) returning the first truthy value - returning the left-side value

      while( value < 5 ){
        value++

      Allows to apply some code right before and while a condition is met.

      var count = 0;
      do {
        count++;
      } while( value < 5 );

      Allows to loop over an Array.
      Careful though, if the condition is never met, this can cause infinite loop.
      Syntax: do {/* ... */ } while(<condition>);

        /* ... */

    • If a code is reading the variable (from a child or root scope ) after the declaring it on the global scope, this would correctly access the value.

    Passing by reference or by value

    Passing by reference or by value concerns how the computer is handling a variable type and how JavaScript will access it - this can relate to the capacity of re-assigning, copy a value to a variable and the variable mutation capabilities

    • By Value:
      Passing by value concerns primitive values that includes: null,undefined, number, boolean, symbol
      • create a value to a unique address in memory
      • copy a value to a unique address in memory without relation
      • mutate a value to a unique address in memory without relation
    • By Reference:
      Passing by reference concerns Objects that includes: Objects,Arrays, function : basically all non-primitive values.
      • create a reference to a unique address in memory
      • copy by re-assigning a variable an object, copies the reference - same address in memory is shared.
      • mutating any related object(s) - which are now bound to the same reference - will mutate all objects.


    How to "copy" a referenced object then?

    This action is called cloning: clones the structure object to assign it to a new reference. When the object needs to be copied and to be created at a new address reference in memory, we need to explicity create a new object. There are two types of cloning: shallow cloning and recursive cloning - commonly called deep cloning. This could put you in a confusing position if you are not aware of of those different behavior and different ways to clone.


    Shallow cloning:
    is the fact to clone an object to a new address in memory ( creating a new reference ); but the nested objects within are not cloned - they are still referenced to the other original object reference.
    This can be done either using: Object.assign({}, <object-variable>)
     or using the spread operator:
    const clonedObj = {...<object-variable>}

    Deep cloning ( Circular cloning ):
    is the fact to clone an object to a new address in memory ( creating a new reference ) including the nested objects - All level of object will be addressed a new reference.
    This can be done either using: JSON.parse(JSON.stringify(<object-variable>))
    which will parse a string representing the object, hence will not have any reference as the returned string is a passed by value
     or using a recent JavaScript global function structureClone():
    const clonedObj = structureClone(<object-variable>, <options>)

    Type Coercion

    Type coercion concerns how JavaScript will interpret the code. Two values being compared but the type is diverging. JavaScript being dynamically typed can handle those unexpected side effects and will attempt to evaluate it for you. It usually occurs when JavaScript attempts to evaluates values together.

    Examples

    • comparison coercion
      • 10 == '10'// true - assuming the within "text" is equal
      • 10 === '10'// false - the type of each value is checked
  • boolean coercion
    • 1 == true
      // true - assuming 1 referring to a binary value which means true
    • 1 === true
      // false - the type of each value is checked
    • 0 == false
      // true - assuming 1 referring to a binary value which means true

    Modules
    Here is how JS got to modules
    • Inline Scripts:
      Script tag with JS in the body
      • lack of code reusability: the script located within a page is not able to be shared between multiple HTML for instance except if it is copy pasted in each HTML file
      • pollutes the Global Namespace: variables and functions declared in the HTML file are exposed globally which led to not being able to use those variables or functions name's again
    • Script tags:
      Script tags importing JS file using the src attribute. (Examples: CDNs can be imported as script tags)
      • Still have to copy paste the script tags between HTML pages that uses them
      • Lacks of Dependency Resolution: Imported scripts sometimes need to be imported in a specific order otherwise they may not work as expected (occurs when a script needs to access code from another script imported but since it has been loaded in an incorrect order: the code cannot be accessed)
      • Still have Global NameScape pollution: namespaces are still exposed and could be used or overridden
    • IIFE:
      Allowing to encapsulate the code to avoid polluting the Global NameSpace and expose it once
      • Does not pollute the Global NameSpace as the previous way - each IIFE exposed uses less Global Name
      • Each IIFE has their own scope
      • Still exposed to the lack of Dependency Resolution problematic
    • Browserify with CommonJS:
      module bundler: running before the site is online and bundle all files modules into one file
      • enabling the syntax: module.export and require(<file-path-with-export>)
        • which reduces again the Global NameSpace
        • which improves the Dependency Resolution
    • Bundler + ES6:
      • ES6:
        import and export or export default keywords allowing to do the same as common JS
      • Bundler (WebPack)
        Bundler such as WebPack, Rollup, Parcel ... are doing the same as browserify but they go through the code checking for ES6 import/export syntaxes
